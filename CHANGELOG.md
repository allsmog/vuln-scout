# Changelog

## Unreleased

- `mobile-exported-component-no-permission` findings now name the component (`android:name`) and, when an intent-filter is present, list every scheme / host / pathPrefix / pathPattern in both the title and `metadata` (`component_name`, `component_kind`, `intent_schemes`, `intent_hosts`, `intent_paths`). Titles are capped at the first 3 entries with a `(+N more)` suffix; full lists stay in metadata for downstream tooling.
- Added `mobile-deeplink-unrestricted` detector: fires on an exported activity that handles a custom-scheme deeplink with no `android:host` / `android:pathPrefix` / `android:pathPattern` filter. http/https app-links are excluded since they're gated by digital asset link verification.
- `_make_finding` now accepts an optional `metadata` dict; empty-string / empty-list / `None` values are dropped at insert time so findings stay compact.
- `mobile_scan.py` now post-processes the merged artifact to enrich the `mobile-nsc-narrow-pinning` finding with the list of hosts the Java/Kotlin code actually contacts but that aren't covered by the manifest's pin-set. The pinning gap finding lives in `apktool_out/`, but the host literals live in `jadx_out/sources/`; the per-target detectors can't see across that boundary, so the merge layer joins them.
- Added `defpackage/` to the third-party path filter. jadx parks all proguard/R8-stripped classes there (Nimbus JOSE, Braintree internals, etc.), and every finding inside it has empirically been third-party SDK code with no user-actionable signal. Dropped real-world decompiled-APK noise by ~67% (21→7 findings on a representative tokenization-flow target) without losing a single calibration anchor.
- Added a `Mobile backup-extractable token chain` chain detector that links `mobile-allow-backup-true` with `mobile-shared-prefs-sensitive`. Either finding alone is "audit later"; together they form a complete adb-backup exfiltration primitive worth surfacing as a chain.
- Added a `Mobile deeplink → in-WebView dispatch chain` chain detector that links an exported deeplink activity (either `mobile-deeplink-unrestricted` or an `mobile-exported-component-no-permission` carrying `intent_schemes` metadata) with any WebView finding (`mobile-webview-js-injection`, `mobile-webview-mixed-content`, `mobile-webview-js-interface`, `mobile-webview-file-access`). Either side alone is incomplete; chained, they describe a complete "land a URL → run attacker JS in privileged native context" primitive.
- `mobile-exported-component-no-permission` now skips framework-provided debug-tooling activities (`androidx.compose.ui.tooling.PreviewActivity` and its `preview.*` sibling). These ship with compose-tooling and only appear in release manifests when the developer forgot to strip the dependency — the app can't patch the component itself, so the finding had no actionable signal.
- `mobile-runtime-exec` now suppresses intentional root-detection probes. Files whose name/class matches `RootDetect`, `checkSu`, `SuExecutable`, `RootBeer`, etc., or whose body contains a hardcoded SU binary literal (`"which su"`, `/system/xbin/su`, `/sbin/su`), are recognized as deliberate anti-tamper checks rather than tainted exec sinks. The vector of concern (untrusted input flowing into `Runtime.exec`) doesn't apply.
- Added a `Mobile permission-relay chain` chain detector that links `mobile-exported-component-no-permission` with `mobile-intent-redirection`. Either finding alone is a hotspot; together they form a complete StrandHogg/Pulse-style permission-relay primitive — any installed app can trampoline an Intent through the exported entry to reach private components or signature-protected actions the attacker could not reach directly.
- `mobile-shared-prefs-sensitive` now names the actual `SharedPreferences` keys being stored in plain text. The title carries the first 3 (e.g. `(key(s): access_token,refresh_token,user_pin)`) and the full list lives in `metadata.pref_keys` for downstream tooling. Previously the finding said only "sensitive value stored" — reviewers had to open the file to learn which token was at risk.
- Added `mobile-cleartext-traffic-allowed` detector for `android:usesCleartextTraffic="true"` at the AndroidManifest application level. Distinct from the existing NSC `cleartextTrafficPermitted` detector — the manifest flag is the global opt-out and is the default on `minSdk<28`. The new finding type is also registered as an MITM-precondition for the existing `Mobile MITM precondition + remote-influenced flow` chain.
- `Mobile backup-extractable token chain` now propagates `metadata.pref_keys` from its participating findings into the chain's impact statement, naming the actual exposed keys (`Exposed key(s): access_token,refresh_token,user_pin`). Combined with the shared-prefs detector enrichment, the chain is now reportable to a triager without opening the underlying findings.
- Markdown report's `## Attack Chains` section now renders the chain `impact` line in addition to `flow_description`. Reviewers previously had to cross-reference the JSON to see the chain's exploit description — the most reviewer-useful field of the chain object.
- Chains now carry a `severity` field set to the max severity across their participants, and the chain list is sorted worst-first. The markdown report renders the severity as a badge next to each chain heading so the highest-impact chain is the first thing a reviewer sees.
- `Mobile deeplink → in-WebView dispatch chain` now also fires on `mobile-deeplink-host-wildcard` (e.g. `<data android:host="*"/>`). A wildcard-host deeplink is strictly more permissive than a scheme-only deeplink — any web page on any host can dispatch the activity — so it should drive the same WebView-XSS chain as the unrestricted variant.
- Chains now carry a deterministic `stable_key` derived from a SHA1 of their name + sorted participant stable_keys. The same chain re-appearing in a later scan hashes to the same key — enabling diff-against workflows to recognize "this chain re-appeared / went away" across runs (the existing `chain-NNN` IDs are sequential and reassigned every scan).
- `mobile-webview-js-injection` no longer carries the parenthetical " (no WebView reference in same file; verify dispatch path)" inside its title. Confidence is now a structured `confidence` field on the finding (`high` when a WebView reference or evaluateJavascript call lives in the same file, `low` when the file only assembles a JS literal). The title stays stable for dedup; the confidence drives severity-and-kind selection in the detector and is available to downstream filters.
- Added a `Mobile token-replay chain` chain detector that links `mobile-shared-prefs-sensitive` with `graphql-sensitive-client-mutation`. Plain-prefs tokens give the attacker the credentials; bundled GraphQL mutations give the attacker the ready-to-fire payload shape. Chain impact names both the exposed pref keys and the at-risk mutation operations so the chain is reportable from the JSON alone.
- Added a `Mobile gadget-landing chain` chain detector that links `mobile-exported-component-no-permission` with `mobile-insecure-deserialization`. Exported entry + ObjectInputStream sink in the same app = Intent-payload-to-gadget-chain RCE shape worth surfacing as a chain (CWE-502 + CWE-926).
- `mobile-exported-component-no-permission` severity now reflects realistic exploit reachability instead of a flat `medium`. Custom-scheme deeplink → high (any web page can dispatch via Intent.parseUri); https/https-only filter → medium (gated by App Links); exported service/receiver/provider → medium (explicit-Intent reachable); exported activity with no intent-filter → low (reachable only by explicit Intent, often accidentally).
- `mobile-log-sensitive` finding now also exposes the matched identifier names as `metadata.sensitive_idents`. The title already named them, but downstream tooling can now pivot on the list programmatically without parsing free-text.
- `submission_template.py` prefers `chain.severity` (set by the chain detector during ranking) over recomputing max-participant severity, and surfaces `chain.stable_key` in the submission header. Submissions can now be cross-referenced across scans by stable_key, and a chain whose severity was deliberately set higher than its participants (e.g. a "critical primitive" with only-medium individual findings) reports its true severity.
- Markdown report's Attack Chains section header now includes the chain's `stable_key` alongside the volatile `chain-NNN` id. Reviewers can search/grep across scans by stable_key without parsing the underlying JSON.
- Per-detector exceptions in `vuln_class_detectors.run_all_detectors` no longer just log silently. The failure list is recorded on `LAST_DETECTOR_FAILURES` and surfaced through `scan_orchestrator` as `tool_statuses['vuln-class-detector'].detector_failures`. The tool status flips from `succeeded` to `degraded` so the output artifact tells operators which sub-detectors crashed without grepping logs.
- `mobile_scan --diff-against PRIOR.json` now also diffs chains by `stable_key`. Output gains `diff.chains.{new,gone,kept}` so reviewers see "the deeplink→WebView chain went away" without comparing individual findings. New-chain entries carry `name` and `severity` so the diff is reportable on its own.
- Markdown report (`--format md`) now renders a `## Changes since prior scan` section right after the executive summary when the artifact carries a diff. Lists new chains, resolved chains, and new findings (top 20) with severity badges. The diff used to live only in the JSON output — reviewers had to grep to see what changed.
- Added iOS chain detectors. `Mobile MITM precondition + remote-influenced flow` chain now also accepts iOS transport gaps (`ios-ats-arbitrary-loads`, `ios-ats-partial-exemption`, `ios-trust-all-ssl`) as preconditions. Added two iOS-only chain patterns: `iOS WebView injection chain` (WKWebView/UIWebView HTML/JS concat + transport gap, CWE-295 + CWE-94) and `iOS credential-at-rest + transport gap chain` (kSecAttrAccessibleAlways keychain item + ATS disabled, CWE-312 + CWE-319).
- Added `Mobile debuggable + in-memory secret chain` chain detector that links `mobile-debuggable-build` with any sensitive-data finding (`mobile-shared-prefs-sensitive`, `mobile-log-sensitive`, `mobile-hardcoded-jwt`, `hardcoded-secret`). Qualitatively different from the backup-extractable chain: process-memory exposure via JDWP attach defeats encryption-at-rest, so even `EncryptedSharedPreferences` and keychain-backed secrets are recoverable once the app pulls them into a String. Chain impact names the at-risk keys/idents propagated from sink metadata.
- `--diff-against` now flags chain severity drift. A chain that survived but escalated or de-escalated lands in `diff.chains.severity_drift` (with from/to severity + direction) instead of getting silently bucketed under `kept`. Markdown report's "Changes since prior scan" section renders the drift list with ↑/↓ arrows so reviewers see "Chain X just escalated medium → critical" without digging into the JSON.
- Chain membership now boosts low-confidence findings to medium with an audit trail (`metadata.confidence_boosted_by_chain`). Reasoning: a `confidence=low` finding (e.g. webview-js-injection with no WebView reference in the same file) is hedged because its dispatch path wasn't locally visible. If chain detection then links that finding to another finding completing the primitive (the WebView lives in a sibling file), the hedge no longer applies — chain context proves the path exists. High/medium confidence findings stay untouched; only the previously-hedged ones get the bump.
- Submission templates now surface participant confidence + the chain-boost audit trail. When a participant's confidence is non-default, the "Affected components" line carries `confidence X`; when `metadata.confidence_boosted_by_chain` is set, it appends `[boosted by chain]`. Reviewers can distinguish organic medium-confidence findings from chain-corroborated promotions when reading the submission.
- Chains now carry a programmatic `pattern` slug (e.g. `mobile-token-replay`, `mobile-backup-exfil`, `ios-webview-injection`) derived from a known-name map with fallback to name-slugification. Downstream consumers (suppressions, alert routers, dashboards) can pivot on `pattern` instead of substring-matching the prose `name`, which is reviewer-facing and may change wording.
- Each finding inside a chain now also carries `chain_pattern` matching its chain's pattern slug. Filters like "ignore all findings inside any mobile-debuggable-secret chain" or "alert on any mobile-token-replay finding" can pivot on a single field on the finding instead of joining the chains array.
- Markdown report's per-finding rows now show the chain pattern slug instead of the volatile chain id (`\[chain mobile-token-replay — sink\]` instead of `\[chain chain-001 — sink\]`). The slug tells the reviewer *what kind of primitive* the finding participates in. When the finding's confidence was promoted by chain context, the confidence line now appends "(boosted by chain)" — same audit trail submission templates already render.
- `.vuln-scout-ignore` now supports chain-pattern suppression. A line `chain_pattern:mobile-debuggable-secret  too noisy on test builds` marks every finding whose `chain_pattern` matches as suppressed (with the trailing text used as `suppression_reason`). Useful for repos where one class of chained primitive is consistently a false positive — e.g. internal dev builds where `debuggable=true` is intentional, or CI sandboxes where NSC pinning is deliberately relaxed.
- Markdown report's "Next Actions" section is now chain-aware. When there are high-or-critical chains, it leads with "Triage the N high-or-critical chain(s) first — start with `<pattern>` (`<stable_key>`)", then suggests `submission_template.py` for bug-bounty writeups, and finally explains how to add `chain_pattern:<slug>` suppressions. Triagers get a workflow that starts with the most impactful primitives, not the historical severity-only ordering.
- `submission_template.py` now names chain submission files by `pattern + stable_key` (e.g. `submission-mobile-token-replay-9a79f7517365.md`) instead of the volatile `submission-chain-001.md`. The same chain across two scans produces the same filename, so submission directories stay organized across re-scans and triage rounds.
- `diff.chains.{new,gone,severity_drift}` entries now expose `pattern` alongside `name`. Markdown report's "Changes since prior scan" section labels chains by their pattern slug (`mobile-token-replay`), not the prose name — exact-match alerting on chain-class changes is now reliable across scans even if the name copy is later edited.
- Chains now carry an aggregated `cwes` array (sorted `CWE-NNN` strings extracted from the chain's `impact` prose plus each participant finding's `message`). Compliance reporting tools, dashboards, and SARIF emitters can pivot on the structured field instead of regex-parsing prose. Markdown report renders linked CWE references in each chain header.
- Submission templates now render `Chain pattern: <slug>` and `CWE: [CWE-XXX](...)` linked references in the header — bug-bounty platforms get clickable CWE links straight from the submission markdown without manual look-up.
- Added `--top-chains N` to `mobile_scan.py`. Keeps only the worst N chains in the output (chains are already sorted worst-first by severity). Findings tagged with dropped chains get their `chain_id`/`chain_role`/`chain_pattern` cleared so downstream tools never dangle references. Designed for CI runs where the triager only wants the highest-impact primitives in the queue.
- Added `Mobile token double-exposure chain` chain detector that links `mobile-shared-prefs-sensitive` with `mobile-log-sensitive` when the *same identifier* shows up in both (`access_token` stored on disk + `accessToken` written to logcat). Identifier matching is normalized — underscores/dashes/case differences across jadx-decompiled naming variants collapse to the same root. Defense-in-depth fails twice: same secret recoverable through disk and log channels independently. (CWE-312 + CWE-532)
- Chains now carry an aggregated `confidence` field set to the *weakest* participant confidence (a chain is only as confident as the finding it depends on most). Aggregation runs AFTER the chain-corroborated confidence boost, so a low-confidence participant that was promoted to medium by chain context puts the chain at medium, not stale-low. Markdown report renders `*(confidence: medium)*` next to the severity badge when the chain is below `high`.
- Submission templates now render `Chain confidence: <level>` in the header — bug-bounty triage can see at a glance whether the chain is high-confidence (file it) or needs more validation first.
- Merged scan `summary` now carries `total_chains`, `chains_by_pattern` (counts keyed by pattern slug), and `chains_by_severity`. Dashboards and SLO reports can chart chain mix and trends across scans without re-walking the `chains` array.
- Markdown executive summary now renders `Chains by Severity` and `Top Chain Patterns` lines using those summary rollups (with fallback to recomputing if the rollups are absent). Reviewers see the app's chain mix at a glance without scrolling to the Attack Chains section.
- Findings can now participate in multiple chains. `chain_id`/`chain_role` stay single-valued for back-compat (captures the first chain that claimed the finding), but every membership accumulates on `chain_participations` = `[{chain_id, role, pattern}, ...]`. Surfaces real shapes: on the Lyft target, `mobile-shared-prefs-sensitive` writes participate in both `mobile-backup-exfil` AND `mobile-token-replay` — previously only the first was visible to downstream filters.
- `chain_pattern:` suppressions now match against *every* entry in `chain_participations[]`, not just the primary `chain_pattern` field. So a finding flagged as a backup-exfil sink that ALSO sits in a token-replay chain can be suppressed via `chain_pattern:mobile-token-replay`. The reason text from the matching rule becomes `suppression_reason`.
- `summary.chains_by_pattern` and `summary.chains_by_severity` are now serialized in a deterministic key order (patterns alphabetically; severities by canonical priority — critical, high, medium, low, info). Same inputs produce byte-identical JSON across re-scans, so CI/CD diffing tools don't churn on key-order noise.
- Chain array ordering is now deterministic within a severity tier. Primary sort: severity desc (worst first, unchanged). Secondary sort: `stable_key` asc — same chains in the same order across runs. Pairs with the chain-rollup determinism (N+44) so the full chains+summary block is byte-stable.
- Markdown report's Hotspots section is now sorted (chain-tagged first, then severity desc, then file) and includes severity badge + chain pattern marker on each line. Triage priority now matches the findings list rather than dumping hotspots in detection order.
- Each finding inside one-or-more chains now carries `chain_cwes` — the deduplicated, sorted union of CWEs aggregated from every chain it participates in. Compliance dashboards can pivot on per-finding CWE lists without rewalking the chains array. A finding that sits in both a backup-exfil chain (CWE-200, CWE-312) AND a token-replay chain (CWE-312) gets `chain_cwes: [CWE-200, CWE-312]`.
- Markdown chain Mermaid diagrams use compact node labels: `[role] type<br/>file:line` instead of the full finding title. Reviewers stop scrolling sideways through wide-node diagrams; the full title still lives in the Findings section. Role is sourced from each chain's `chain_participations` entry — so multi-chain findings render with the right per-chain role in each diagram.
- SARIF emitter now propagates `chain_cwes` into rule tags as `external/cwe/cwe-NNN` so GitHub Code Scanning surfaces them as filterable rule tags. Related-location walking also follows every chain in `chain_participations[]` (not just the primary `chain_id`), so multi-chain findings link to all related findings in the SARIF output. Pairs with N+42 multi-chain participation + N+47 chain_cwes propagation.
- Chains now carry `cvss_estimate` — a numeric CVSS-3.1 band midpoint derived from severity (critical=9.5, high=8.0, medium=5.5, low=3.0). Named `cvss_estimate` (not `cvss_score`) so consumers know it's a band centroid, not a per-vector calculation. Useful for SARIF `security-severity`, numeric CI gates, and dashboards.
- Markdown chain header now renders `*(CVSS≈8.0)*` next to severity + confidence so reviewers see the numeric chain score inline without opening the JSON.
- `apply_suppressions` now recomputes chain-level rollups after applying suppressions. Chains whose participants are *all* suppressed get marked `chain.suppressed=true` and excluded from `total_chains` / `chains_by_pattern` / `chains_by_severity`. Closes the gap where `chain_pattern:` suppression rules would silence findings but the dashboards still showed the chain count.
- Markdown report's Attack Chains, Executive Summary, and Next Actions all skip `chain.suppressed=true` chains. Reviewer-facing report stays clean; chains remain in the JSON for audit history.
- `submission_template.py` now skips suppressed chains. Operators using `chain_pattern:` suppression rules no longer have to clean up unwanted submission files manually — the generator respects the suppression intent end-to-end.
- `--diff-against` filters out suppressed chains on both prior and current sides before bucketing into new/gone/kept/severity_drift. A suppressed chain present in current is treated as "gone" from the diff's perspective — matching the suppression semantics now applied uniformly across rollups, markdown report, submissions, and diff.
- SARIF `relatedLocations` now also skips links via suppressed chains. Operators silencing a chain class no longer see the silenced cross-finding link graph resurface in GitHub Code Scanning / SARIF consumers.
- `chain_pattern:` suppression rules now support trailing wildcards. `chain_pattern:mobile-*` silences every chain whose pattern starts with `mobile-`; `chain_pattern:ios-*` silences the iOS family. One line in `.vuln-scout-ignore` per family, instead of one per slug.
- `summary` now tracks `suppressed_chains` (total) and `suppressed_chains_by_pattern` (pattern→count). Dashboards can show "5 active, 2 suppressed (mobile-debuggable-secret: 2)" without re-walking the chains array, and operators can verify a suppression rule is actually firing.
- Added `mobile_scan.py --list-chain-patterns`. Prints every known chain pattern slug with its human-readable name, sorted alphabetically. `.vuln-scout-ignore` authors no longer have to grep `chain_detector.py` to discover the slugs available for `chain_pattern:<slug>` suppression rules.
- `--list-chain-patterns` also surfaces the wildcard prefixes for dynamic web/service chains (`ssrf-to-*`, `auth-bypass-to-*`, `path-traversal-to-*`). These chains have per-scan slugs (e.g. `ssrf-to-command-injection`), so the wildcard pattern is the right way to silence them in `.vuln-scout-ignore`.
- Added a regression test pinning the expected set of chain-pattern slugs returned by `list_chain_patterns()`. Future edits that drop or rename a slug now break the test rather than silently breaking the operator's `.vuln-scout-ignore` references.
- Added a CLI smoke test that invokes `mobile_scan.py --list-chain-patterns` via subprocess. Catches the case where the function is fine but the CLI wiring (sys.argv short-circuit, exit-before-arg-parse) regresses.
- Updated `/vuln-scout:mobile-audit` (and the deprecated `/whitebox-pentest:mobile-audit`) argument-hints to advertise `--top-chains N` and `--list-chain-patterns`. Release checklist requires argument-hint to include every flag the command supports — those flags shipped in N+37 and N+59 but the docs had drifted.
- `/vuln-scout:mobile-audit` command body now documents the new flags with usage examples and shows the full `.vuln-scout-ignore` syntax — per-finding stable_key, exact chain_pattern, and prefix-wildcard chain_pattern rules. Operators reading the slash-command help see the complete suppression workflow without digging through the changelog.
- Added `--list-chain-patterns-format {text,json}`. Default `text` keeps the operator-facing aligned-column output; `json` emits a clean `[{"pattern": ..., "name": ...}]` array for tooling, CI scripts, and dashboards. Argument-hint updated on both command copies.
- Each of the 16 chain patterns now runs in isolation via a `_safe()` wrapper. A buggy or in-development pattern detector previously crashed the whole chain pipeline; now individual failures log at WARNING and skip, and the remaining patterns still produce chains. Mirrors the per-detector resilience added to `vuln_class_detectors.run_all_detectors` in N+20.
- Chain pattern failures are now also recorded on `chain_detector.LAST_PATTERN_FAILURES` and surfaced in the merged artifact as `scan_metadata.chain_pattern_failures` = `[{"pattern": "<slug>", "error": "<msg>"}, ...]`. Operators see which chain patterns crashed this scan without grepping logs — pairs N+66's resilience with N+20's detector_failures audit trail.
- `scan_orchestrator` (the main `/scan` pipeline) now also captures `chain_detector.LAST_PATTERN_FAILURES` after running chains, and `build_artifact` threads it through into `scan_metadata.chain_pattern_failures`. Visibility is consistent across both `mobile_scan` and `scan_orchestrator` entry points.
- Added defensive `isinstance(p, dict)` guards across three call sites that iterate `chain_participations[]` — chain pattern propagation, chain-pattern suppression, SARIF related-locations. Hand-edited or externally-produced artifacts with malformed (None / non-dict) entries no longer crash the pipeline.
- Added a regression test that fires every mobile/iOS chain pattern and asserts each produced chain has a `pattern` slug present in the static `_CHAIN_NAME_TO_PATTERN` map. Catches the case where a chain detector's `name` field is silently renamed, leaving the pattern slug to fall back to slug derivation and breaking operator `chain_pattern:` rules.
- Added `mobile_scan.py --no-chains`. Skips chain detection entirely — output contains findings but `chains: []` and zero chain-aware enrichments. Designed for ultra-quick CI smoke scans where the operator only wants the raw finding list, or for benchmarking the chain-detection overhead in isolation.
- `--no-chains` + `--top-chains` is now rejected (`parser.error("--no-chains is incompatible with --top-chains")`) — they're logically incoherent. `--no-chains` + `--diff-against` warns but still runs (finding-level diff still works; chain-level diff fields are simply empty).
- `/vuln-scout:mobile-audit` command body now includes usage examples for `--no-chains` (ultra-quick smoke scan) and `--list-chain-patterns --list-chain-patterns-format json` (tooling integration). The argument-hint had advertised both flags but the body had no examples — operators now see them inline.
- `chain_pattern:` suppression rules now tolerate whitespace after the colon. Programmatic callers passing `{"chain_pattern: mobile-*": "..."}` (with leading space in the slug) get the same behavior as the canonical `chain_pattern:mobile-*` form. The slug part is `lstrip()`'d on extraction.
- Added `file:<glob>` suppression syntax. Rules like `file:*/test/*  test code, not shipped` or `file:build/generated/*  auto-generated code` silence every finding whose `file` matches the fnmatch glob. Closes a long-standing gap where operators had to enumerate every individual finding to silence a whole directory. Priority order is now: stable_key → legacy_key → file-glob → chain_pattern (first match wins).
- Verified that `file:<glob>` suppression composes with the chain-rollup recompute from N+52: when every participant in a chain is file-glob-suppressed, the chain itself gets `chain.suppressed=true` and drops out of `chains_by_pattern` / `chains_by_severity` / `total_chains`. Suppression cascade is now consistent across all four rule types (stable_key, legacy_key, file-glob, chain_pattern).
- `chain_pattern:` suppression rules now accept full fnmatch globs (not just trailing `*`). `chain_pattern:*-webview-*` matches both `mobile-webview-dispatch` and `ios-webview-injection`. Exact-slug matches still take precedence before any glob is checked.
- `file:<glob>` rules with empty or whitespace-only patterns (e.g. `file:` or `file:   `) are now silently dropped before glob matching, instead of becoming an empty pattern that could match every file. Defensive against operator typos in hand-edited `.vuln-scout-ignore` files.
- Same empty-rule defense applied to `chain_pattern:` rules. `chain_pattern:` or `chain_pattern:   ` (empty/whitespace-only slug) are now dropped — previously they would silently suppress every chain because `"" in any_chain_pattern` evaluates True in the prefix-match path.
- "Changes since prior scan" markdown section now sorts new chains and new findings by severity desc (with stable_key/file:line as secondary keys for byte-stable diffs). Reviewers see the worst new chain/finding at the top of each list — pairs with N+10's worst-first chain ordering for consistency.
- Chain severity drift list sorts escalations (↑) before de-escalations (↓), then by `to_severity` desc — the most-urgent drift leads. Resolved-chains list sorts by `stable_key` for byte-stable output across runs.
- Added an end-to-end test pinning the diff-section rendering: canonical section order (new chains → resolved chains → drift → new findings → resolved findings), critical-before-medium sort within new chains/findings, and escalations-before-de-escalations within drift. Future regressions in section order or sort logic break the test first.
- `apply_suppressions` always resets `summary.total_chains` / `chains_by_pattern` / `chains_by_severity` to safe defaults (`0` / `{}` / `{}`) when the artifact has no chains. Previously these fields would keep stale pre-suppression values, confusing dashboards.
- Added `severity:<level>` suppression rule. `severity:low  CI ignores noise` silences every finding at-or-below `low` (i.e. `low` + `info`). Useful for CI workflows that only care about high+ findings. Multiple `severity:` rules take the highest floor (most aggressive). Priority order across all rule types: stable_key → legacy_key → severity → file-glob → chain_pattern.
- `/vuln-scout:mobile-audit` command body example block now documents `severity:<level>` and middle-wildcard `chain_pattern:*-webview-*` rules + the full priority order. Closes the doc gap that's been growing since N+75 (file-glob), N+77 (full globs), N+85 (severity-floor).
